# aterm Research Dossier - Overview

This is the decision-grade synthesis of twelve domain research docs ([01](01-warp-internals.md) through [12](12-licensing.md)). It states the cross-cutting architecture, resolves the one gating decision (the render stack), summarizes the other key decisions with links, and consolidates risks and owner questions. It is written to be built from by engineers and AI agents. Where a domain doc and the adversarial verification disagree, the corrected fact is used here and flagged.

## What aterm is (one paragraph)

aterm is a native, GPU-rendered macOS terminal in Rust that clones the *behavior* of Warp (a controlled native UI wrapping a hidden background shell over a PTY, rendering its own block-based timeline rather than a raw VT grid) but with the radically minimal visual language of iA Writer. It spawns a real login shell over a PTY, parses the byte stream itself, and segments output into one-grid-per-command blocks driven by shell-integration hooks. A single shell-first input box, with a hotkey that flips where Enter routes (live shell vs the AI agent) while preserving the typed text, feeds either the shell or a full agentic loop (plan -> act -> observe -> repeat) backed by Anthropic Claude, with a deterministic code-side risk gate, OS-level sandboxing, and a rich transcript that lives in the same single wall-clock timeline as the command blocks. The headline non-functional requirement, against which everything is judged, is a guaranteed 60fps floor for normal use (typing, scrolling, streaming output), ideally 120fps on ProMotion. It is GPLv3 and macOS-first, architected so Linux/Windows are not precluded.

## Cross-cutting architecture (the spine)

The system is three layers behind a one-way dependency arrow, plus two leaf crates. The crate split mirrors the proven prototype and is enforceable in CI ([10](10-packaging-scaffold.md)):

```
aterm-app  ->  aterm-agent  ->  aterm-core
   |                                  ^
   +--> aterm-tokens (leaf) ----------+      aterm-bench (off to the side, deps core)
```

- **aterm-core** - the engine, no UI, no LLM. PTY spawn/resize/signals, VT parsing + grid, the block model, OSC-133/OSC-7 mark interception, the shell-shim extraction. This is where the 60fps story lives or dies.
- **aterm-agent** - the Claude Messages-API client, the agentic turn loop, the streaming-event mapper, the deterministic risk gate, the single Secrets source, and the output sanitizer. Pure logic, heavily unit-testable, no window.
- **aterm-app** - the binary `aterm`. The window, the GPU renderer, the unified timeline UI, the input box + hotkey routing, the approval UI. The only crate that touches the OS window/GPU and the only one packaged.
- **aterm-tokens** - design tokens (colors, spacing, type scale, font names) as typed Rust, a leaf so it never creates a cycle.
- **aterm-bench** - the Criterion / iai-callgrind harnesses for the perf floor.

**Threading model** (convergent across every fast native terminal and validated by [03](03-pty-vt-rust.md) and [09](09-performance-60fps.md)): three threads communicating over bounded mailboxes.

1. **PTY reader thread** - blocking `read()` into a ~64 KiB reusable buffer, sends bytes over a *bounded* channel (bounded = implicit backpressure; under a `cat`/`yes` flood the reader blocks on send, which blocks read, which lets the kernel PTY buffer apply flow control). Never touches the GPU.
2. **Model thread** - owns the `Term`, the grid, and the `BlockList`. Drains the channel, runs bytes through the OSC-133/7 pre-parser filter, then the VT parser, mutates blocks, and publishes an immutable snapshot + dirty regions. **Coalesces** bursts on a ~4-8ms tick so a megabyte burst becomes one parse pass + one frame, not thousands.
3. **Render thread** - driven by the display vsync callback, reads the latest snapshot, draws at 60/120fps, never blocks on the model or the PTY.

**Data flow.** PTY bytes -> OSC mark filter (strips our marks to zero-width, tags offsets, decides alt-screen suppression at fire-time) -> VT parser -> grid mutation -> block segmentation on OSC-133 A/B/C/D -> immutable per-block row snapshots on command finish -> SumTree height index for O(log n) viewport queries -> snapshot to renderer. The agent runs on a tokio runtime *off* the render thread; SSE deltas land on the UI by channel and mutate the current timeline entry incrementally (never relaying out the whole timeline per delta). Agent-run commands are ordinary terminal blocks; agent conversation steps are block variants in the *same* list, sorted by wall-clock timestamp. This single-timeline, one-block-primitive design is the structural reason human and agent activity interleave with no special-casing - confirmed as Warp's actual model ([01](01-warp-internals.md), [06](06-agent-architecture.md)) and verified.

**The block boundary signal is hooks, not VT inference.** Command boundaries come from shell-integration hooks (preexec/precmd) emitting marks, installed via per-shell env-var indirection (zsh ZDOTDIR shim, bash ENV/--rcfile, fish XDG_DATA_DIRS) - never by editing user dotfiles. aterm consumes OSC 133 (A/B/C/D) + OSC 7 as the canonical lifecycle, nonce-tags every mark to defeat nested-shell spoofing, and surfaces a visible three-state integration indicator (Integrated / Heuristic / None) instead of the prototype's silent zsh-only degradation ([04](04-shell-integration.md)).

## THE RENDER-STACK RECOMMENDATION (the gating decision)

**Recommendation: build aterm's UI/render layer as a custom `wgpu` + `parley`/`cosmic-text` renderer (the "Warp path"), behind a thin internal `aterm-ui` seam, driven by a self-bridged `CADisplayLink` loop. Do NOT adopt GPUI as the UI framework. Confidence: medium-high.**

This reverses the primary recommendation of the render-stack doc ([02](02-render-stack-eval.md), which led with GPUI) and aligns with the licensing ([12](12-licensing.md)) and competitive ([11](11-competitive-landscape.md)) researchers. The synthesis judgment is that GPUI's three independent liabilities - GPL contamination, zero accessibility, and a pre-1.0 dependence on a competitor's framework - collectively outweigh its real "months of saved build time" advantage for a project whose entire identity is a custom, owned, minimal, openly-licensed terminal.

### Scored-matrix summary

The render-stack doc's own weighted matrix (gating weight on the 60fps floor) scored GPUI ~62 and the custom wgpu+parley path ~57 out of 80 - a *4-point gap that turns almost entirely on time-to-first-pixel (GPUI wins) vs accessibility (custom wins)*. Both stacks score 5/5 on raw perf headroom: GPUI has Zed's shipping 120fps proof, and the custom path has Warp's shipping >144fps / ~1.9ms-redraw proof (both vendor figures, not independently reproduced). egui (~46) and iced (~40) are disqualified for broken IME and absent accessibility; vello (~43) is alpha. So the real contest is two-horse, and the matrix margin is inside the noise of three factors the matrix under-weighted: licensing permanence, strategic independence, and the a11y gap.

### Rationale (why custom wgpu over GPUI)

1. **Licensing - the decisive new fact.** GPUI declares Apache-2.0 but a default release build statically links GPL-3.0-or-later object code through `gpui -> sum_tree -> ztracing -> zlog/ztracing_macro` (Zed issue #55470, open with no fix - verified). For a GPLv3 app this is not a *violation*, but it (a) permanently forecloses any future relicense/dual-license/permissive-fork, and (b) makes aterm track someone else's accidental copyleft on every bump. The custom stack (wgpu, winit, parley/cosmic-text, swash - all MIT and/or Apache-2.0, verified) is unambiguously clean.
2. **Accessibility.** GPUI has no working VoiceOver/screen-reader support for the editing surface, and AccessKit-into-GPUI is an open effort "far beyond Zed 1.0" (verified). parley already integrates AccessKit text properties. If a11y is ever a requirement, GPUI is disqualified outright; the custom path ships it.
3. **Strategic independence and product fit.** aterm's wedge post-Warp-open-source is *feel + openness + agent integration*, not "blocks in Rust" ([11](11-competitive-landscape.md)). Owning the renderer - the terminal grid fast-path, the block/timeline layout, the input box - is the product, and it removes a dependence on a fast-moving, officially-under-supported-outside-Zed, pre-1.0 framework built by an adjacent competitor.
4. **The perf floor is an architectural property we must own anyway.** Per [09](09-performance-60fps.md), 60fps is won by vsync-driven rendering, damage tracking, PTY/render decoupling, and zero per-frame allocation - all of which we implement ourselves regardless of framework. GPUI would hand us Zed's macOS frame-pacing lessons for free, but those lessons are *documented* (corrected: Zed kept CVDisplayLink and fixed pacing by presenting drawables early + a ~1s keep-warm window, rather than reverting to CADisplayLink as one source claimed) and reproducible. We adopt CADisplayLink as the safe default loop driver and inherit the present-early + keep-warm discipline by reading the writeup, not by importing the framework.

The text pipeline is the same either way ([08](08-text-glyph-rendering.md)): GPU glyph atlas + alpha-only mask + single instanced draw call, CoreText rasterization on macOS, grayscale AA only, one HarfRust shaping engine with two layout front-ends (constant-advance grid + proportional prose). The custom path uses cosmic-text 0.19 + swash + glyphon (or parley) for this; GPUI would have used its own CoreText path. Choosing custom keeps the terminal grid fast-path fully under our control.

### Strongest counter-argument

GPUI is the *only* Rust stack that has already shipped aterm's exact hard problem - a 120fps GPU scene-graph mixing a monospace editor grid with proportional prose - in a real product (Zed). Choosing the custom path means rebuilding, from scratch, the entire retained UI layer: layout, hit-testing, focus, event routing, the input box, the selection model, IME plumbing, and the block/transcript widgets. That is weeks-to-months of work GPUI would give us on day one, and every line of it is a chance to reintroduce a bug GPUI already fixed. For a small team, time-to-first-pixel is itself an existential risk, and "we own it" is cold comfort if the project never ships. This counter-argument is real and is the reason confidence is medium-high, not high.

### What would change the decision (toward GPUI)

- A binding decision that aterm will *never* relicense or permit permissive/commercial forks (removes liability #1), **and**
- A decision that screen-reader accessibility is permanently out of scope or acceptably deferred indefinitely (removes #2), **and**
- A 1-2 week GPUI spike (mono grid + input box + virtualized transcript + CJK IME on gpui 0.2.2 + the bundled Nerd Font) that confirms standalone fps, IME quality, and PUA-glyph atlas behavior - none of which are independently verified today.

If all three land, GPUI's time-to-first-pixel advantage would justify reversing this call. **Recommended gate either way: fund the 1-2 week render spike ([09](09-performance-60fps.md) open question 1) to validate winit 0.30 + self-bridged CADisplayLink + the wgpu present path against the frame-time budget before locking.**

## Other key decisions (1-2 lines each)

- **Terminal engine** ([03](03-pty-vt-rust.md)): `alacritty_terminal` 0.26 for the VT core (grid, reflow, alt-screen, damage, `RenderableContent`) - the same core Zed's terminal uses, though via Zed's own git fork, so depend on the published crate directly (corrected). `portable-pty` 0.9 for the cross-platform PTY (keeps the Windows door open). We intercept OSC 133/7 ourselves because alacritty does not parse them (issue #5850, verified). Finished blocks store immutable row snapshots to dodge alacritty's known reflow sharp edges.
- **Input model** ([05](05-unified-input-ux.md)): one pure-Rust `InputModel` reducer holding text + selection + a `mode: Shell|Agent` field; the hotkey mutates only `mode`, so text is preserved by construction (fixing the prototype's worst sin). ropey 0.6 (or plain String) for v1. IME via winit 0.30 `Ime` events with a hand-rolled `NSTextInputClient` as the escape hatch; a `preedit-active` gate is top priority so Enter confirms an IME candidate and never submits. Mode shown by caret tint + prompt glyph, no banner.
- **Agent loop** ([06](06-agent-architecture.md)): call the Claude Messages API directly over HTTP from a thin typed Rust client (reqwest + tokio + SSE + serde); reject the Agent SDK (Python/TS-only, bundles a Claude Code binary, Commercial-Terms-licensed - conflicts with GPLv3; all verified). Typed custom tools (run_command/read_file/edit_file/list_dir/glob/grep), argv not shell, every call gated. Port the prototype's deterministic code-side risk gate + single Secrets source + output sanitizer nearly verbatim. Back the classifier with a real boundary: macOS Seatbelt via `sandbox-exec` (deprecated but the only documented mechanism, used by Anthropic's own sandbox-runtime - verified) behind a `Sandbox` trait, plus setrlimit + timeout-kill.
- **Shell integration** ([04](04-shell-integration.md)): consume OSC 133 + OSC 7 as canonical; ingest OSC 633/1337 opportunistically. Full bash + fish from day one, version-branched. bash hooks via `PS0='${ preexec;}'` + PROMPT_COMMAND on bash 5.3+, else the bundled bash-preexec shim (which supports **bash 3.1+** - corrected from "4.0-5.2"; this matters because macOS's frozen 3.2 is covered). Nonce-tag every mark; suppress blocks on the alt screen at fire-time; never degrade silently.
- **Text + glyph stack** ([08](08-text-glyph-rendering.md)): GPU glyph atlas + alpha mask + single instanced draw call; CoreText rasterization; grayscale AA only; ASCII grid bypass that skips shaping for plain runs; a built-in sprite face for box-drawing/Powerline/braille and a generated Nerd Font per-codepoint constraint table to fix PUA-glyph misalignment. Bundle the Duo/Quattro proportional variants (only Mono is currently vendored).
- **Design system** ([07](07-ia-design-language.md)): two themes day one (warm "paper" light + dark) from one hue family so UI and ANSI output cohere; three-register fonts (Mono NFM grid / Duo prose / Quattro chrome); one blue accent used scarcely; risk-gate verdicts in the iA syntax-highlight hue family, always color + text label; motion capped to 3 animations <=220ms to protect the frame floor. The exact accent blue is derived, not source-verified - confirm before locking tokens.
- **Performance proof** ([09](09-performance-60fps.md)): a two-tier CI-gated harness - iai-callgrind instruction-count micro-benches gating every PR (noise-immune on shared runners), plus an in-process frame recorder running 7 scripted stress scenarios nightly on real Apple Silicon ProMotion hardware with p50/p99/max/dropped-frame and separate input-latency gates. The hard gate is the 60fps floor; 120fps is tracked.
- **Packaging + scaffold** ([10](10-packaging-scaffold.md)): five-crate virtual workspace, cargo-packager 0.11.x for .app + .dmg, winit-level hidden titlebar, OFL fonts via `ATSApplicationFontsPath`, signing deferred to a later milestone, jj-based landing workflow and `docs/tickets` + `docs/adr` + CLAUDE.md conventions ported from the prototype.
- **Licensing** ([12](12-licensing.md)): GO for GPLv3. Permissive render/PTY/text stack is clean; iM Writing Nerd Font is bundleable under SIL OFL 1.1 (ship OFL text + iA/IBM copyright, keep the "iMWriting" name); BYOK with Keychain storage satisfies Anthropic's Commercial Terms. Add `cargo deny check licenses` with a GPL/AGPL denylist. Treat Warp's now-AGPL client as study-only, never copy source.
- **Competitive position** ([11](11-competitive-landscape.md)): Warp open-sourced its client under AGPL-3.0 in April 2026 (verified), so the wedge is no longer "blocks + AI in Rust" but iA-minimalism + single shell-first input + full GPLv3 openness + deep agent integration, plus being the best *host* for Camp B agents (Claude Code/Codex) via MCP auto-discovery and OSC-133. Ghostty is the latency/throughput bar for the hidden shell layer (community latency is middle-of-pack ~13ms, not a 2-5ms leader - corrected).

## Risk register (top risks, ranked, with mitigations)

1. **The 60fps floor is unproven on our actual stack.** All cited fps figures (Warp >144fps/1.9ms, Zed 120fps) are vendor blog claims on their hardware/content, not reproduced. winit's macOS redraw timing is historically awkward and bridging our own CADisplayLink may need unsafe AppKit interop; wgpu's informal 5-10% overhead (one maintainer's estimate, not a benchmark - corrected) sits on the critical path. *Mitigation: fund the 1-2 week render spike before locking; renderer trait with a `metal`-crate fallback; the CI frame-recorder harness as the standing proof.*
2. **Choosing custom wgpu over GPUI lengthens time-to-first-pixel by weeks-to-months.** We rebuild the whole retained UI layer ourselves. *Mitigation: the thin `aterm-ui` seam keeps the GPUI option reversible; lean on parley's `PlainEditor` + IME + AccessKit and glyphon's atlas rather than building text from zero; ship a vertical slice early.*
3. **The risk gate is a classifier, not a security boundary.** Residual RCE via an un-enumerated interpreter or novel shell construct is possible; prompt-injection defense is layered, not absolute. *Mitigation: Seatbelt sandbox is mandatory not optional, behind a trait; setrlimit + timeout-kill; ask-always default with session-scoped auto-run; single Secrets source feeds both gate and sanitizer so they cannot drift.*
4. **alacritty_terminal is pre-1.0 (0.26) with breaking minors and known live-grid reflow correctness/perf issues** (jumbling, scrollback loss, slow reflow on large windows). *Mitigation: pin exactly; re-verify Handler/Event signatures against the pinned version before coding; immutable per-block snapshots make history immune to reflow; perf-check a maximized 4K resize early.*
5. **Shell integration is fragile at the edges.** OSC-133 is load-bearing and alacritty won't parse it (our filter must handle split sequences and BEL-vs-ST exactly); env-var indirection (ZDOTDIR/ENV/XDG_DATA_DIRS) may not survive exec/su/sudo/login; prompt frameworks (starship/p10k) may double-fire marks; bash 3.2 DEBUG-trap reliability is unbenchmarked. *Mitigation: nonce-gate every mark (drops un-nonced framework marks); visible three-state indicator; real test matrix across shells/versions; degrade loudly, never silently.*
6. **`sandbox-exec` is deprecated with no published Apple removal date.** *Mitigation: `Sandbox` trait so a future native-API/VM backend swaps in; resource limits + the gate remain even if Seatbelt is pulled.*
7. **No official Rust Anthropic SDK; we own a hand-rolled client and track API drift** (beta headers, server-tool version bumps). *Mitigation: thin client typed against a small, stable surface; pin `anthropic-version: 2023-06-01`; community crates as reference only.*
8. **Bundled iMWriting + custom font path is partly untested.** Only the Mono variant is currently vendored (Duo/Quattro missing); PUA Nerd Font glyphs need a per-codepoint constraint table (real, easily under-scoped work); CJK IME quality on our IME path is unverified. *Mitigation: add Duo/Quattro to the bundle; generate the constraint table from the Nerd Fonts patcher (license-check any vendored Ghostty data); spike CJK IME early.*
9. **Design tokens are partly unverified taste.** The accent blue hex is derived (not sampled from iA), WCAG ratios are computed estimates, ANSI tuning is hand-derived, and light "paper" + bright ANSI is the riskiest combo. *Mitigation: sample/confirm the accent, validate contrast with a real library, eyeball ANSI against real tools (ls/vim/htop/git diff) on both themes before locking.*
10. **CI cannot prove "60fps always" without dedicated hardware.** Shared GitHub macOS runners are non-ProMotion and too noisy for frame gating. *Mitigation: iai-callgrind micro-gate on shared runners every PR; a self-hosted Apple Silicon ProMotion runner for nightly Tier-2 (an infra commitment the owner must approve).*

## Open questions for the product owner (deduped across domains)

These are the decisions only the human can make; they are ordered by how much downstream work they gate.

1. **Render stack: approve the custom-wgpu recommendation, or keep GPUI in play pending a spike?** This is the gating decision. Tied sub-questions: is screen-reader accessibility a launch requirement (forces custom) or deferrable? Must aterm preserve the option to relicense later (forces custom)? Is depending on a competitor's pre-1.0 framework strategically acceptable? Will you fund the 1-2 week render spike before locking? ([02](02-render-stack-eval.md), [09](09-performance-60fps.md), [12](12-licensing.md))
2. **Is real-time multi-user collaboration ever in scope?** Warp baked an operation-based CRDT into the editor from day one; if collaboration is out, aterm skips that complexity entirely (recommended: out). ([01](01-warp-internals.md), [05](05-unified-input-ux.md))
3. **Threads vs tokio for PTY I/O?** Drives `portable-pty` (blocking, cross-platform) vs `pty-process` (async, Unix-only) and must align with the runtime/render choice. And: is Windows in scope for v1 at all, or only "not precluded later"? ([03](03-pty-vt-rust.md))
4. **Agent autonomy default and the destructive-command set the gate must always confirm.** Confirm ask-always as the shipped default (recommended). Plus: network egress policy for agent commands (deny-all + allowlist vs allow + proxy-log, which drives the Seatbelt profile); API-key custody (Keychain recommended); single-agent loop vs coordinator + Haiku sub-agents in v1; Anthropic-only vs multi-provider for v1. ([06](06-agent-architecture.md))
5. **Shell matrix and degradation policy.** Minimum shell version floor (support macOS system bash 3.2 degraded, or require Homebrew bash 5.x?); SSH/Docker subshell integration in v1 or deferred (recommended deferred, shown honestly as "remote - no integration"); honest "no blocks" raw view vs a labeled heuristic block UI for un-integrated shells. ([04](04-shell-integration.md))
6. **Mode-toggle hotkey and agent-turn input policy.** Confirm the default toggle key (proposal Cmd-/, rejecting Ctrl-Space and Cmd-.) and whether it is rebindable day one; keep the input box live during agent turns (queue next message, Esc interrupts) vs the prototype's full swallow. Confirm Opt-Enter stays a mode-independent one-shot send-to-agent. ([05](05-unified-input-ux.md))
7. **Product narrative emphasis.** Is the flagship the built-in Claude agent, or being the best terminal to *host* any agent? How explicitly do we attack Warp ("the open, minimal Warp")? Is aterm strictly local-first / no-account as a stated value? ([11](11-competitive-landscape.md))
8. **Design defaults.** First-launch theme (follow macOS appearance vs default to "paper" light); how loud the risk gate is (quiet chip vs full-width banner); honor terminal-app color/caret requests (DECSCUSR, OSC palette) verbatim vs enforce the aterm theme; confirm or sample the accent blue. ([07](07-ia-design-language.md))
9. **Build + release logistics.** Bundle identifier (proposal `ai.ameba.aterm`); minimum macOS version (proposal 11.0); Rust edition (2021 recommended); whether to own a self-hosted Apple Silicon ProMotion CI runner for nightly frame benchmarks; the milestone at which signing/notarization becomes a requirement. ([09](09-performance-60fps.md), [10](10-packaging-scaffold.md))
10. **Counsel for a one-time pre-GA review** of (a) the GPLv3/AGPL boundary vs Warp and (b) the Anthropic BYOK monetization question, if aterm ever monetizes. ([12](12-licensing.md))

## Recommended build order (epics, high level)

The ordering front-loads the two existential risks (the perf floor and the engine) and defers everything orthogonal to shipping a fast, correct, block-rendering terminal.

1. **Epic 0 - Scaffold + render spike (de-risk before committing).** Five-crate workspace, mise tasks, CI (fmt/clippy/test), jj/docs conventions. In parallel, the 1-2 week render spike: winit 0.30 + self-bridged CADisplayLink + wgpu present path + a triangle/glyph through the atlas, measured against the frame budget. **Gate: lock the render stack only after the spike.**
2. **Epic 1 - Terminal core + GPU grid.** `alacritty_terminal` + `portable-pty`, the 3-thread reader/model/render split with output coalescing, the GPU glyph atlas + grid fast-path, damage tracking, the keep-warm window. The Tier-1 iai-callgrind benches land here. **Goal: a fast raw terminal that holds the 60fps floor under `cat`/`yes`/scroll - the foundation everything sits on.**
3. **Epic 2 - Shell integration + block model.** ZDOTDIR/ENV/XDG_DATA_DIRS shims for zsh/bash/fish, the OSC-133/7 pre-parser filter with nonce gating, the `BlockList` + SumTree height index, immutable per-block snapshots, the visible integration indicator, alt-screen pass-through. **Goal: command/output blocks in one timeline.**
4. **Epic 3 - Unified input box.** The pure `InputModel` reducer with the `mode` field, IME via winit (NSTextInputClient escape hatch), the routing brain with the preedit-active gate, async/debounced highlight + ghost text, the hotkey toggle preserving text, the iA mode indicator. **Goal: the one shell-first input with preserved-text routing.**
5. **Epic 4 - Design system pass.** Wire `aterm-tokens` (two themes, three font registers, ANSI palettes), the component specs (command block, prompt, agent card, status chip, risk-gate badge), bundle Duo/Quattro, the Nerd Font constraint table + sprite face. **Goal: the iA look, validated on real tool output.**
6. **Epic 5 - Agent loop + safety.** The thin Claude Messages client (SSE off-thread), typed tools, the ported risk gate + single Secrets source + sanitizer, Seatbelt + setrlimit, the timeline transcript model, the approval UX + autonomy controls. **Goal: full agentic loop in the same timeline, gated and sandboxed.**
7. **Epic 6 - MCP + interop.** The Messages-API MCP connector for remote servers, a local stdio MCP client, MCP config auto-discovery so aterm hosts Camp B agents well; route every MCP call through the gate. **Goal: best host for external agents.**
8. **Epic 7 - Tier-2 perf harness + hardening.** The in-process frame recorder, the 7 scripted stress scenarios on a self-hosted ProMotion runner, input-latency gates, the resize/reflow perf check, edge-case shell matrix testing.
9. **Epic 8 (deferred) - Packaging, signing, polish.** cargo-packager .app/.dmg, hidden titlebar, OFL bundle + acknowledgements UI, signing/notarization when distribution matters, Focus-Mode analog, completions menus.
